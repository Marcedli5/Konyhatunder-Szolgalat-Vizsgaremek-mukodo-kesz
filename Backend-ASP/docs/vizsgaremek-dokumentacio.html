Forras: a projekt jelenlegi specifikacioja es a veglegesitett adatbazis-terv alapjan osszeallitott dokumentacio.

1. Projekt attekintes

A vizsgaremek celja a Konyhatunder Szolgalat weboldalanak teljeskulso es belso megujitasa. A projekt egyszerre foglalkozik a latogatok szamara lathato modern webes felulettel, a regisztralt felhasznalok rendelest tamogato funkcioival, valamint az adminisztrativ feladatokat ellato backend es asztali kornyezet megvalositasaval.

Fo celok
  • A weboldal megjelenesenek modernizalasa es atlathatobb felhasznaloi elmeny kialakitasa.
  • Felhasznaloi regisztracio es bejelentkezes bevezetese.
  • Online rendelesi folyamat tamogatasa visszaigazolassal.
  • Adminisztratori etlap- es rendeleskezeles kialakitasa.
  • Bizalmas adatbazis-adatok biztonsagos tarolasa es vedelme.

Regi design referencia: https://www.konyhatunderszolgalat.hu

  • ASP .NET: backend szolgaltatasok es szerveroldali logika

  • React Native: frontend fejlesztes
  • MySQL: adatbazis-kezelorendszer

4. Celcsoportok es fo funkciok

A rendszer harom jol elkulonitheto felhasznaloi csoport kore szervezodik: latogatok, bejelentkezett felhasznalok es adminisztratorok. Minden csoport mas celra hasznalja az oldalt, ezert az elerheto funkciok is eltérnek.

4.1 Latogatok

A latogatok azok a felhasznalok, akik meg nem rendelkeznek fiokkal, vagy eppen nem jelentkeztek be. Szamukra az oldal elsodleges szerepe a tajekozodas, a szolgaltatas bemutatasa es az etlap megtekintese.

  • Fooldal megtekintese
  • Aktualis etlap megtekintese bejelentkezes nelkul
  • Rolunk oldal megnyitasa
  • Kapcsolati adatok megtekintese
  • Rendeles inditasanak lehetosege a bejelentkezes vagy regisztracio fele terelve
  • Bejelentkezo es regisztracios felulet elerese

4.2 Bejelentkezett felhasznalok

A regisztralt es bejelentkezett felhasznalok a latogatoi funkciokon felul mar a szolgaltatas aktiv hasznaloi. Szamukra a rendszer fo erteke a kenyelmes rendelesi folyamat es a sajat adataik kezelese.

  • Minden latogatoi funkcio elerese
  • Online rendeles leadasa
  • Rendelesrol visszaigazolas fogadasa
  • Sajat profil adatok megtekintese es frissitese
  • Korabbi es aktualis rendelesek listazasa
  • Az etlap teteleinek reszletes, tapanyag-adatokat is tartalmazo megtekintese

4.3 Adminisztratorok

Az adminisztratorok a rendszer uzemeltetoi. Olyan belso funkciokra van szukseguk, amelyekkel az etlapot ossze tudjak allitani, a megrendeleseket ellenorizni tudjak, es a szolgaltatas napi mukodeset iranyitani tudjak.

  • Etlap szerkesztese
  • Etlap nyomtathato valtozatainak eloallitasa
  • Rendelesek attekintese, darabszamok kovetese
  • Rendelok helyenek vagy cimenek megtekintese
  • Szolgaltatoi oldalon a megrendeles reszleteinek kezelese
  • Felhasznalokkal kapcsolatos ugyintezes es hibajegyek kezelese

5. Reszletes funkcionális specifikacio

5.1 Hitelesites es felhasznalokezeles

  • A felhasznalok regisztralni tudnak nevvel, e-mail-cimmel, telefonszammal es lakhelyhez kapcsolodo adatokkal.
  • A felhasznalok be tudnak jelentkezni a sajat fiokjukba.
  • A jelszavak nyers formaban nem tarolhatok, csak biztonsagos hash formaban.
  • A szerepkorok kezelese lehetove teszi a latogatoi, felhasznaloi es adminisztratori szintek szetvalasztasat.

5.2 Rendeleskezeles

  • A bejelentkezett felhasznalo az aktualis etlap alapjan rendelest adhat le.
  • A rendszer visszaigazolast kuld a megrendelonek.
  • A rendeleshez tartozik statusz, datum, megjegyzes es a megrendelt menuk darabszama.
  • Az egyes rendelesek arainak torteneti helyesseget a rendeles tetelein tarolt egysegar biztositja.

5.3 Etlap- es menukezeles

  • Az adminisztrator napi etlapot tud letrehozni.
  • Egy etlaphoz tobb menu kapcsolhato.
  • Egy menu tobb etelbol allhat, peldaul levesbol, foetelbol es desszertbol.
  • Az etlap minden felhasznaloi csoport szamara lathato.
  • Az etelek mellett kcal, feherje, zsir, szenhidrat, so es allergen informaciok is megjelennek.

5.4 Nyomtatas es tobbfele megjelenes

Az etlap nyomtatasahoz opcionálisan harom kulonbozo vizualis megjelenes tamogathato:

  • Solar
  • MenuMano
  • Konyhatunder Szolgalat

Ez a funkcio kulonosen hasznos, ha a szolgaltatas kulonbozo formaban szeretne megjeleniteni ugyanazt a napi etlapot.

5.5 Hibajegykezeles

  • A felhasznalok hibajegyet vagy problemabejelentest tudnak rogzitani.
  • A hibajegy tipushoz, leirashoz, allapothoz es letrehozas idejehez kotodik.
  • Az adminisztrator a hibajegyek allapotat kovetni es modositani tudja.

6. Biztonsagi kovetelmenyek

  • A bizalmas adatbazis-adatok vedelme kotelezo.
  • A jelszavak tarolasa kizarolag hash formaban tortenhet.
  • Az e-mail-cim nem lehet idegen kulcs helyett altalanos kapcsolati azonositó.
  • A jogosultsagkezelest szerepkorokkel vagy jol meghatarozott hozzaferesi szintekkel kell megvalositani.
  • A felhasznaloi es adminisztratori funkcioknak el kell kulonulniuk.

7. Adatbazis-terv ertekelese

A kiindulasi adatbazisterv jo alapot adott, mert tartalmazta a fo uzleti elemeket: felhasznalok, rendelesek, menuk, etelek, kategoriak es hibajegyek. Ugyanakkor tobb ponton javitasra szorult a normalizacio, a biztonsag es a bovitheto strukturak erdekeben.

7.1 Fo problemak a korabbi tervben

  • Az e-mail-cim kapcsolati kulcskent szerepelt, ami valtozhat es uzletileg nem stabil.
  • A telefon INT tipussal szerepelt, ami nem alkalmas valos telefonszamok helyes tarolasara.
  • A jelszotarolas plain text vagy gyenge megoldas fele mutatott.
  • Az etelek hozzavaloit kulon oszlopokban tarolta a terv, ami nem bovitheto.
  • A menuk es etlapok kozotti kapcsolat tul merev volt, mert egy menu egyetlen etelhez kotodott.
  • Hianyoztak fontos allapot- es naplozo mezok, peldaul statusz, created_at es updated_at.

7.2 Alkalmazott javitasok

  • Minden kapcsolat elsodlegesen id alapu idegen kulcsokra epul.
  • A felhasznalokat es szerepkoroket az ASP Identity sajat tablai kezelik.
  • Az alapanyagok es receptek kulon tablaba kerultek.
  • A rendelesek es rendelesi tetelek szet lettek valasztva.
  • Az etlap naphoz kotott publikacios strukturat kapott.
  • Beepult az allergenek kezelese.

8. Veglegesitett adatbazis-struktura

Az alabbi szerkezet a projekt jelenlegi, veglegesitett adatbazis-megoldasat irja le. A kod az eredeti anyag alapjan lett rendezve, es egy szukseges hibat is javit: az allergens tabla tartalmazza a code oszlopot, mivel az egyedi megszoritas erre hivatkozik. A felhasznalokat es szerepkoroket az ASP Identity kezeli, ezert a sajat users, roles es user_roles tablakat a terv mar nem tartalmazza.

8.1 Fo entitasok

  • ASP Identity (AspNetUsers, AspNetRoles, ...): hitelesites es jogosultsagkezeles
  • categories, foods: etelkategoriak es etelek
  • allergens, food_allergens: allergen kezeles
  • ingredients, recipe_items: alapanyagok es receptkapcsolatok
  • menus, menu_items: menuk es azok tetelei
  • daily_menus, menu_availability: napi etlap es elerhetoseg
  • orders, order_items: rendelesek es rendelesi tetelek
  • ticket_types, tickets: hibajegyek

8.2 Kapcsolatok

  • AspNetUsers 1:N orders
  • orders 1:N order_items
  • menus 1:N order_items
  • menus 1:N menu_items
  • foods 1:N menu_items
  • categories 1:N foods
  • foods 1:N recipe_items
  • ingredients 1:N recipe_items
  • daily_menus 1:N menu_availability
  • menus 1:N menu_availability
  • AspNetUsers 1:N tickets
  • ticket_types 1:N tickets
  • A szerepkorok es a felhasznalo-szerepkor kapcsolatok az ASP Identity reszei.

8.3 Vegleges adatbazis kod

-- Alap beallitas CREATE DATABASE IF NOT EXISTS vizsgaremek_etlap CHARACTER SET utf8mb4 COLLATE utf8mb4_hungarian_ci; USE vizsgaremek_etlap; -- Az AspNetUsers / AspNetRoles es kapcsolodo Identity tablakat az alkalmazas sajat migracioi hozzak letre. CREATE TABLE categories ( id INT UNSIGNED PRIMARY KEY AUTO_INCREMENT, name VARCHAR(40) NOT NULL, CONSTRAINT uq_categories_name UNIQUE (name) ) ENGINE=InnoDB; CREATE TABLE foods ( id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT, name VARCHAR(200) NOT NULL, category_id INT UNSIGNED NOT NULL, kcal DECIMAL(7,2) NULL, protein_g DECIMAL(7,2) NULL, fat_g DECIMAL(7,2) NULL, carbs_g DECIMAL(7,2) NULL, salt_g DECIMAL(7,2) NULL, created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP, CONSTRAINT fk_foods_category FOREIGN KEY (category_id) REFERENCES categories(id) ON DELETE RESTRICT, INDEX ix_foods_category (category_id), INDEX ix_foods_name (name) ) ENGINE=InnoDB; CREATE TABLE allergens ( id SMALLINT UNSIGNED PRIMARY KEY AUTO_INCREMENT, code VARCHAR(20) NOT NULL, name VARCHAR(120) NOT NULL, description VARCHAR(255) NULL, CONSTRAINT uq_allergens_name UNIQUE (name), CONSTRAINT uq_allergens_code UNIQUE (code) ) ENGINE=InnoDB; CREATE TABLE food_allergens ( food_id BIGINT UNSIGNED NOT NULL, allergen_id SMALLINT UNSIGNED NOT NULL, PRIMARY KEY (food_id, allergen_id), CONSTRAINT fk_food_allergen_food FOREIGN KEY (food_id) REFERENCES foods(id) ON DELETE CASCADE, CONSTRAINT fk_food_allergen_allergen FOREIGN KEY (allergen_id) REFERENCES allergens(id) ON DELETE RESTRICT, INDEX ix_food_allergens_allergen (allergen_id) ) ENGINE=InnoDB; CREATE TABLE ingredients ( id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT, name VARCHAR(120) NOT NULL, base_unit ENUM('g','kg','ml','l','db') NOT NULL, CONSTRAINT uq_ingredients_name UNIQUE (name) ) ENGINE=InnoDB; CREATE TABLE recipe_items ( food_id BIGINT UNSIGNED NOT NULL, ingredient_id BIGINT UNSIGNED NOT NULL, amount DECIMAL(10,3) NOT NULL, unit ENUM('g','kg','ml','l','db') NOT NULL, PRIMARY KEY (food_id, ingredient_id), CONSTRAINT fk_recipe_food FOREIGN KEY (food_id) REFERENCES foods(id) ON DELETE CASCADE, CONSTRAINT fk_recipe_ingredient FOREIGN KEY (ingredient_id) REFERENCES ingredients(id) ON DELETE RESTRICT, CHECK (amount > 0) ) ENGINE=InnoDB; CREATE TABLE menus ( id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT, code CHAR(1) NOT NULL, price_ft INT UNSIGNED NOT NULL, is_active TINYINT(1) NOT NULL DEFAULT 1, created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP, CONSTRAINT uq_menus_code UNIQUE (code), CHECK (price_ft > 0) ) ENGINE=InnoDB; CREATE TABLE menu_items ( menu_id BIGINT UNSIGNED NOT NULL, food_id BIGINT UNSIGNED NOT NULL, course_order TINYINT UNSIGNED NOT NULL, PRIMARY KEY (menu_id, food_id), CONSTRAINT fk_menu_items_menu FOREIGN KEY (menu_id) REFERENCES menus(id) ON DELETE CASCADE, CONSTRAINT fk_menu_items_food FOREIGN KEY (food_id) REFERENCES foods(id) ON DELETE RESTRICT, INDEX ix_menu_items_menu (menu_id), INDEX ix_menu_items_food (food_id), CHECK (course_order >= 1) ) ENGINE=InnoDB; CREATE TABLE daily_menus ( id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT, menu_date DATE NOT NULL, note VARCHAR(200) NULL, created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, CONSTRAINT uq_daily_menus_date UNIQUE (menu_date) ) ENGINE=InnoDB; CREATE TABLE menu_availability ( daily_menu_id BIGINT UNSIGNED NOT NULL, menu_id BIGINT UNSIGNED NOT NULL, max_qty INT UNSIGNED NULL, PRIMARY KEY (daily_menu_id, menu_id), CONSTRAINT fk_menu_avail_daily FOREIGN KEY (daily_menu_id) REFERENCES daily_menus(id) ON DELETE CASCADE, CONSTRAINT fk_menu_avail_menu FOREIGN KEY (menu_id) REFERENCES menus(id) ON DELETE RESTRICT ) ENGINE=InnoDB; CREATE TABLE orders ( id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT, user_id VARCHAR(255) NOT NULL, order_date DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, status ENUM('draft','placed','preparing','ready','cancelled') NOT NULL DEFAULT 'placed', comment VARCHAR(200) NULL, CONSTRAINT fk_orders_user FOREIGN KEY (user_id) REFERENCES AspNetUsers(Id) ON DELETE RESTRICT, INDEX ix_orders_user_date (user_id, order_date), INDEX ix_orders_status (status) ) ENGINE=InnoDB; CREATE TABLE order_items ( order_id BIGINT UNSIGNED NOT NULL, menu_id BIGINT UNSIGNED NOT NULL, qty INT UNSIGNED NOT NULL DEFAULT 1, unit_price_ft INT UNSIGNED NOT NULL, PRIMARY KEY (order_id, menu_id), CONSTRAINT fk_order_items_order FOREIGN KEY (order_id) REFERENCES orders(id) ON DELETE CASCADE, CONSTRAINT fk_order_items_menu FOREIGN KEY (menu_id) REFERENCES menus(id) ON DELETE RESTRICT, CHECK (qty > 0), CHECK (unit_price_ft > 0) ) ENGINE=InnoDB; CREATE TABLE ticket_types ( id SMALLINT UNSIGNED PRIMARY KEY AUTO_INCREMENT, name VARCHAR(80) NOT NULL, CONSTRAINT uq_ticket_types_name UNIQUE (name) ) ENGINE=InnoDB; CREATE TABLE tickets ( id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT, user_id VARCHAR(255) NOT NULL, ticket_type_id SMALLINT UNSIGNED NOT NULL, description TEXT NOT NULL, status ENUM('open','in_progress','closed') NOT NULL DEFAULT 'open', created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP, CONSTRAINT fk_tickets_user FOREIGN KEY (user_id) REFERENCES AspNetUsers(Id) ON DELETE RESTRICT, CONSTRAINT fk_tickets_type FOREIGN KEY (ticket_type_id) REFERENCES ticket_types(id) ON DELETE RESTRICT, INDEX ix_tickets_user_status (user_id, status), INDEX ix_tickets_type_status (ticket_type_id, status) ) ENGINE=InnoDB;

9. Adatbazis-tervezesi megjegyzesek

  • Az etlap mindenki szamara lathato, beleertve a vendegeket is.
  • Az etelek tapanyagadatai kotelezo dokumentacios elemek.
  • Az allergen-informaciok miatt kulon kapcsolat lett bevezetve a foods es allergens tablák kozott.
  • A jelenlegi modellek a projektben mar Class Library formaban, scaffoldinggal lettek letrehozva a Models mappaban.

10. Tovabbi fejlesztesi lehetosegek

  • E-mailes vagy alkalmazason beluli automatikus visszaigazolasok bovítese.
  • Rendelesi allapotok valos ideju kovetese.
  • Nyomtatasi sablonok reszletesebb testreszabasa.
  • Statisztikai kimutatasok adminisztratorok szamara.
  • Profilkezeles tovabbi adatokkal, peldaul szallitasi cim vagy preferenciak tarolasaval.
